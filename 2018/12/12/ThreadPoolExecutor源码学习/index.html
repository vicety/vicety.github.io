异步调用实现

上面的代码涉及了几个类/接口：ThreadPoolExecutor、ExecutorService、Executors、Future，下面列一下这些类的作用以及继承结构

  • ThreadPoolExecutor：线程池的实现类，ThreadPoolExecutor -> AbstractExecutorService -> ExecutorService -> Executor
  • ExecutorService：定义了一个可执行服务应有的功能，如shutDown、submit、invokeAll等
  • Executor：只定义了一个execute(Runnable)方法
  • Executors：内部提供静态方法以及静态类，用于创建预定义的ExecutorService
  • Future：定义了一个异步计算应该实现的功能，包括get、cancel、isCancelled、done。Future对象可以看做是任务执行结果的一个暂存位置，通过get函数可以阻塞取得计算结果。

然后分析代码执行
submit函数传入Runnable，

  • RunnableFuture：同时继承Future与Runnable的接口
  • FutureTask：在其中存储异步操作的当前阶段、计算得到的结果、等待结果的线程（下方总结有更详细的分析）

newTaskFor返回一个FutureTask对象，然后交由实现类实现的execute方法执行

使用get函数获得异步计算的结果，如果结果还没出来，那么调用awaitDone函数，这里考虑没有超时机制的简单情况，就是将线程挂载到等待链表的头结点，成功后使当前线程休眠（可能有多个线程都阻塞在这个future的get方法上）

有休眠必然有唤醒，看一下unpark在哪里，ctrl+f发现在finishCompletion方法中从链表表头遍历，对每个线程进行unpark操作，最后调用子类重写的done回调（done默认为空）

finishCompletion有三处出现，分别是set、cancel、setException，对应三种退出方式，其中set函数的唯一调用在run方法（run方法又是在ThreadPoolExecutor的Worker线程中被调用的），提交的任务在这里被调用，运行计算得到结果后调用set方法将运算结果写入Object类型成员变量，随后将所有等待执行结果的、阻塞在awaitDone方法的线程unpark（finishCompletion方法）。

值得注意的是任务在运行时若产生异常，setException方法将状态转换为EXCEPTION，异常本身存储到outcome中，若未发生异常，set方法设置状态为NORMAL，方法返回值存储到这个变量。get方法调用时根据FutureTask的状态返回Future泛型类型的outcome或者将outcome转型为Throwable后抛出异常。

总结一下：AbstractExecutorService负责将用户通过submit方法提交的任务（Callable、Runnable）包装为FutureTask，提交到线程池的不再是用户任务，而是包装后的FutureTask。FutureTask实现三个功能：

  1. 包装用户提交的Runnable，为其存储计算结果、异常，为其存储等待计算结果的线程（WorkNode）、计算结束后唤醒所有等待者
  2. 提供get方法作为阻塞获取计算结果的API（也包含若任务未完成则挂起加入等待队列的逻辑）、isDone/isCanceled作为非阻塞获取计算结果的API。
  3. 通过volatile state原子地维护任务状态，在get时根据状态返回结果或抛出异常。

以上解答了异步调用如何被实现，接下来看线程池如何管理线程

线程管理

线程池中定义了如下的状态：

  • RUNNING：普通的运行中状态
  • SHUTDOWN：不再接受新任务
  • STOP：不再接受新任务，放弃等待队列中的任务，interrupt正在执行任务的线程
  • TIDYING：所有的任务都已经终止，workerCount=0，运行回调方法terminated
  • TERNIMATED：回调方法terminated运行完毕后的状态

线程池的构造方法可以指定一些参数，Executors中预定义的线程池给出了默认参数

构造方法参数说明：

  • corePoolSize：核心线程数，默认情况下核心线程会一直存活，即使处于闲置状态也不会受存keepAliveTime限制。除非将- allowCoreThreadTimeOut设置为true。
  • maximumPoolSize：线程池所能容纳的最大线程数。超过这个数的线程将被阻塞。当任务队列为没有设置大小的LinkedBlockingDeque时，这个值无效。
  • keepAliveTime：非核心线程的闲置超时时间，超过这个时间就会被回收。
  • unit：指定keepAliveTime的单位，如TimeUnit. SECONDS。当将allowCoreThreadTimeOut设置为true时对corePoolSize生效。
  • workQueue：线程池中的任务队列. 常用的有三种队列
  • a. SynchronousQueue：是一种无缓冲的等待队列，在某次添加元素后必须等待其他线程取走后才能继续添加；
  • b. LinkedBlockingDeque：是一个无界缓存的等待队列，不指定容量则为Integer最大值，锁是分离的；
  • c. ArrayBlockingQueue：是一个有界缓存的等待队列，必须指定大小，锁是没有分离的；
  • threadFactory：线程工厂，提供创建新线程的功能，通过线程工厂可以对线程的一些属性进行定制。
  • RejectedExecutionHandler：当线程池达到maximumPoolSize且阻塞队列已满，会调用RejectedExecutionHandler的rejectedExecution方法，也就是某种拒绝策略，默认有以下四种。
    • AbortPolicy：当任务添加到线程池中被拒绝时，它将抛出RejectedExecutionException 异常。
    • CallerRunsPolicy：当任务添加到线程池中被拒绝时，会使用提交此任务的线程处理被拒绝的任务。
    • DiscardOldestPolicy：当任务添加到线程池中被拒绝时，线程池会放弃等待队列中最旧的未处理任务，然后将被拒绝的任务添加到等待队列中。
    • DiscardPolicy：当任务添加到线程池中被拒绝时，线程池将丢弃被拒绝的任务。

从线程池为空的状态开始，到来的任务首先与一个新线程一起包装为Worker加入workers集合，如果workers.size达到corePoolSize则加入阻塞队列，如果队长有限，那么队满后加入的任务继续包装为Worker加入workers集合，直到达到maximumPoolSize，此后加入的任务全部采用拒绝策略拒绝

线程状态与线程数量使用一个int存放，高三位存状态，其余位存worker数量

可以看出Worker是内部重要的执行单元，Worker继承了AQS，重写tryAcquire、tryRelease、lock、unlock等方法实现为一个不可重入的排它锁；Worker内部存储一个Thread和一个Task作为任务执行的环境；创建Worker时会利用ThreadFactory创建一个对应此Worker的线程

异步调用是如何实现的？

首先创建线程池，执行submit(Callable)方法。那么执行submit(Callalbe)操作发生了什么？

1. 创建一个RunnableFuture类的FutureTask，交由线程池执行
2. 线程池根据线程状态
    1. 创建新线程执行
    2. 入队响应空闲线程对阻塞队列的take操作或者等待线程空闲
    3. 根据拒绝策略拒绝执行这一任务，默认的拒绝策略有：
        1. 抛异常
        2. 什么也不做，将任务丢弃
        3. 交给调用submit的线程执行
        4. 将队头元素poll抛弃，对此任务执行execute函数

在异步任务计算的同时，调用线程可以做一些别的事情，需要计算结果时调用get方法获得结果。那么调用get获得执行结果时发生了什么？

  • FutureTask.run会调用内部的Callable，将正常调用返回的结果或是异常存储在一个Object对象中，随后通过unpark唤醒所有内部维护的、正在等待这个计算结果的线程。
  • FutureTask的run方法会根据结果是否已经产生修改FutureTask的状态，如果状态不合适（比如说结果还没计算出来或者计算还没开始），调用方会加入等待队列，同时通过yield或者park让出CPU，直到下次轮询（yield的场景）或FutureTask包装的Runnable完成后将等待着一一unpark

FutureTask可以总结为如下几点：

  1. 包装用户的Runnable，为任务执行结果（或异常）分配了一个暂存空间
  2. 实现Future接口，在用户侧可以阻塞/非阻塞地检查任务结果
  3. 为阻塞获取结果的接口(get)实现结果未出时的挂起，以及结果出现时的unpark通知
  4. 通过volatile int实现任务状态的可见性

线程池如何管理线程？

  • 线程池中的每个Worker都对应一个线程，当任务到来时

    • if(workers.size < corePoolsize) 创建新的worker
    • else if(worker.size == corePoolsize && queue.notFull) 任务入队
    • else if(worker.size <= maximumPoolSize && queue.full) 创建新的worker
    • else 根据拒绝策略拒绝执行这一任务

  • 对于任务的执行过程无论是否发生异常都不影响方法返回

  • 每个worker都执行一个死循环，不停地调用getTask方法从阻塞队列中获得任务然后执行，worker有两种退出情况

    • 执行过程中抛出异常（在默认情况下FutureTask会接住所有异常从而不会在这一层收到异常）导致线程死亡，另外线程死循环外finally中的processWorkerExit回考虑是否需要在线程死亡后补上一个新线程
    • getTask方法返回null，原因有
      • 线程池状态改变
      • 超过maximumPoolSize
      • 超过corePoolSize但不超过maximumPoolSize的线程空闲太久导致timeout

其他问题

  • 为什么Worker本身也要继承AQS设计为不可重入的排它锁？

    • 一种原因是用于在interruptIdleWorkers函数中判断一个Worker是否空闲，因为在运行时是需要加锁的。
    • 至于为什么不继承ReentrantLock，见StackOverflow上的讨论 （我觉得这个面试肯定不问……）

  • 线程池参数如何设置：简单地讨论需要的线程个数的话，对于CPU密集型任务设为CPU核心数+1，对于IO密集型设为CPU核心数*N，IO耗时越久，这个N可以越大
