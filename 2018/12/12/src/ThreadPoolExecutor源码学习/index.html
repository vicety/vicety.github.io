基本用法

先学习一下ThreadPool的基本使用

public static void main(String[] args) throws InterruptedException, ExecutionException {
    long l = System.currentTimeMillis();
    ExecutorService executorService = Executors.newSingleThreadExecutor();
    Future<Integer> future = executorService.submit(() -> {
        System.out.println("执行耗时操作");
        try {
            Thread.sleep(2500);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("执行结果就绪");
        return 100;
    });

    mainThreadTasks();
    System.out.println("到达阻塞点");
    System.out.println(String.format("future执行结果为：%d", future.get()));
    System.out.println(String.format("执行时间：%d", System.currentTimeMillis() - l));
}

对于上述代码提出两个问题：

  1. 异步调用是如何实现的？
  2. 线程池如何管理线程？

异步调用实现

上面的代码涉及了几个类/接口：ThreadPoolExecutor、ExecutorService、Executors、Future，下面列一下这些类的作用以及继承结构

  • ThreadPoolExecutor：线程池的实现类，ThreadPoolExecutor -> AbstractExecutorService -> ExecutorService -> Executor
  • ExecutorService：定义了一个可执行服务应有的功能，如shutDown、submit、invokeAll等
  • Executor：只定义了一个execute(Runnable)方法
  • Executors：内部提供静态方法以及静态类，用于创建预定义的ExecutorService
  • Future：定义了一个异步计算应该实现的功能，包括get、cancel、isCancelled、done

然后分析代码执行
submit函数传入Runnable，

// AbstractExecutorService.java
public <T> Future<T> submit(Callable<T> task) {
    if (task == null) throw new NullPointerException();
    RunnableFuture<T> ftask = newTaskFor(task);
    execute(ftask);
    return ftask;
}
  • RunnableFuture：同时继承Future与Runnable的接口
  • FutureTask：在其中存储异步操作的当前阶段、计算得到的结果、等待结果的线程

newTaskFor返回一个FutureTask对象，然后交由实现类实现的execute方法执行

FutureTask中定义了一个异步计算的生命周期如下：

/*
 * Possible state transitions:
 * NEW -> COMPLETING -> NORMAL
 * NEW -> COMPLETING -> EXCEPTIONAL
 * NEW -> CANCELLED
 * NEW -> INTERRUPTING -> INTERRUPTED
 */
private volatile int state;
private static final int NEW          = 0;
private static final int COMPLETING   = 1;
private static final int NORMAL       = 2;
private static final int EXCEPTIONAL  = 3;
private static final int CANCELLED    = 4;
private static final int INTERRUPTING = 5;
private static final int INTERRUPTED  = 6;

从状态名就可以看出状态代表的含义，NORMAL指普通的执行结束

使用get函数获得异步计算的结果，如果结果还没出来，那么调用awaitDone函数，这里考虑没有超时机制的简单情况，就是将线程挂载到等待链表的头结点，成功后使当前线程休眠（可能有多个线程都阻塞在这个future的get方法上）

public V get() throws InterruptedException, ExecutionException {
    int s = state;
    if (s <= COMPLETING) // 还没出结果
        s = awaitDone(false, 0L);
    return report(s);
}

private V report(int s) throws ExecutionException {
    Object x = outcome; // x可能是返回值或异常
    // 如果调用正常返回
    if (s == NORMAL)
        return (V)x;
    // 如果是cancelled/interrupting/interrupted
    if (s >= CANCELLED)
        throw new CancellationException();
    // 如果调用过程产生异常
    throw new ExecutionException((Throwable)x);
}

private int awaitDone(boolean timed, long nanos)
    throws InterruptedException {
    final long deadline = timed ? System.nanoTime() + nanos : 0L;
    WaitNode q = null;
    boolean queued = false;
    for (;;) {
        if (Thread.interrupted()) {
            removeWaiter(q);
            throw new InterruptedException();
        }

        int s = state;
        // 运行完毕，返回FutureTask状态用于对运算结果的转型（为Throwable或FutureTask本身的泛型）
        if (s > COMPLETING) {
            if (q != null)
                q.thread = null;
            return s;
        }
        else if (s == COMPLETING) // 仍在计算，让出CPU
            Thread.yield();
        // 否则运算还没开始，插入Wait链表，最终park挂起线程
        else if (q == null)
            q = new WaitNode();
        else if (!queued)
            queued = UNSAFE.compareAndSwapObject(this, waitersOffset,
                                                    q.next = waiters, q);
        else if (timed) {
            nanos = deadline - System.nanoTime();
            if (nanos <= 0L) {
                removeWaiter(q);
                return state;
            }
            LockSupport.parkNanos(this, nanos);
        }
        else
            LockSupport.park(this);
    }
}

有休眠必然有唤醒，看一下unpark在哪里，Ctrl+f发现在finishCompletion方法中从链表表头遍历，对每个线程进行unpark操作，最后调用子类重写的done回调（done默认为空）

private void finishCompletion() {
    // assert state > COMPLETING;
    for (WaitNode q; (q = waiters) != null;) {
        if (UNSAFE.compareAndSwapObject(this, waitersOffset, q, null)) {
            for (;;) {
                Thread t = q.thread;
                if (t != null) {
                    q.thread = null;
                    LockSupport.unpark(t);
                }
                WaitNode next = q.next;
                if (next == null)
                    break;
                q.next = null; // unlink to help gc
                q = next;
            }
            break;
        }
    }

    done();

    callable = null;        // to reduce footprint
}

finishCompletion有三处出现，分别是set、cancel、setException，对应三种退出方式，其中set函数的唯一调用在run方法（run方法又是在ThreadPoolExecutor的Worker线程中被调用的），提交的任务在这里被调用，运行计算得到结果后调用set方法将运算结果写入Object类型成员变量，随后将所有等待执行结果的、阻塞在awaitDone方法的线程unpark。

值得注意的是任务在运行时若产生异常，setException方法将状态转换为EXCEPTION，异常本身存储到outcome中，若未发生异常，set方法设置状态为NORMAL，方法返回值存储到这个变量。get方法调用时根据FutureTask的状态返回Future泛型类型的outcome或者将outcome转型为Throwable后抛出异常。

protected void set(V v) {
    if (UNSAFE.compareAndSwapInt(this, stateOffset, NEW, COMPLETING)) {
        outcome = v;
        UNSAFE.putOrderedInt(this, stateOffset, NORMAL); // final state
        finishCompletion();
    }
}

public void run() {
    if (state != NEW ||
        !UNSAFE.compareAndSwapObject(this, runnerOffset,
                                        null, Thread.currentThread()))
        return;
    try {
        Callable<V> c = callable;
        if (c != null && state == NEW) {
            V result;
            boolean ran;
            try {
                result = c.call();
                ran = true;
            } catch (Throwable ex) {
                result = null;
                ran = false;
                setException(ex);
            }
            if (ran)
                set(result);
        }
    } finally {
        // runner must be non-null until state is settled to
        // prevent concurrent calls to run()
        runner = null;
        // state must be re-read after nulling runner to prevent
        // leaked interrupts
        int s = state;
        if (s >= INTERRUPTING)
            handlePossibleCancellationInterrupt(s);
    }
}

总结一下：​ThreadPoolEexecutor接收callable，包装为FutureTask，插入ThreadPoolEexecutor的阻塞队列等待执行，ThreadPoolEexecutor的Worker（对应一个线程）从阻塞队列取出FutureTask，调用其run方法，run方法内部调用之前传入的callable，计算获得结果后存储在FutureTask的Object类型成员变量中等待需要的线程来取，当然也有可能已经有线程正在等待，此时FutureTask通过Unsafe.unpark唤醒内部维护的等待线程链表中的所有线程，这些线程从awaitDone方法中解除阻塞，根据任务执行的状态码返回执行结果或是抛出执行过程中的异常或是抛出执行取消异常。

以上大致解答了异步调用如何被实现，接下来看线程池如何管理线程

线程管理

线程池中定义了如下的状态：

  • RUNNING：普通的运行中状态
  • SHUTDOWN：不再接受新任务
  • STOP：不再接受新任务，放弃等待队列中的任务，interrupt正在执行任务的线程
  • TIDYING：所有的任务都已经终止，workerCount=0，运行回调方法terminated
  • TERNIMATED：回调方法terminated运行完毕后的状态

线程池的构造方法可以指定一些参数，Executors中预定义的线程池给出了默认参数

public ThreadPoolExecutor(int corePoolSize,
                          int maximumPoolSize,
                          long keepAliveTime,
                          TimeUnit unit,
                          BlockingQueue<Runnable> workQueue,
                          ThreadFactory threadFactory,
                          RejectedExecutionHandler handler) {
    if (corePoolSize < 0 ||
        maximumPoolSize <= 0 ||
        maximumPoolSize < corePoolSize ||
        keepAliveTime < 0)
        throw new IllegalArgumentException();
    if (workQueue == null || threadFactory == null || handler == null)
        throw new NullPointerException();
    this.corePoolSize = corePoolSize;
    this.maximumPoolSize = maximumPoolSize;
    this.workQueue = workQueue;
    this.keepAliveTime = unit.toNanos(keepAliveTime);
    this.threadFactory = threadFactory;
    this.handler = handler;
}

构造方法参数说明：

  • corePoolSize：核心线程数，默认情况下核心线程会一直存活，即使处于闲置状态也不会受存keepAliveTime限制。除非将- allowCoreThreadTimeOut设置为true。
  • maximumPoolSize：线程池所能容纳的最大线程数。超过这个数的线程将被阻塞。当任务队列为没有设置大小的LinkedBlockingDeque时，这个值无效。
  • keepAliveTime：非核心线程的闲置超时时间，超过这个时间就会被回收。
  • unit：指定keepAliveTime的单位，如TimeUnit.SECONDS。当将allowCoreThreadTimeOut设置为true时对corePoolSize生效。
  • workQueue：线程池中的任务队列.常用的有三种队列
  • a.SynchronousQueue：是一种无缓冲的等待队列，在某次添加元素后必须等待其他线程取走后才能继续添加；
  • b.LinkedBlockingDeque：是一个无界缓存的等待队列，不指定容量则为Integer最大值，锁是分离的；
  • c.ArrayBlockingQueue：是一个有界缓存的等待队列，必须指定大小，锁是没有分离的；
  • threadFactory：线程工厂，提供创建新线程的功能，通过线程工厂可以对线程的一些属性进行定制。
  • RejectedExecutionHandler：当线程池达到maximumPoolSize且阻塞队列已满，会调用RejectedExecutionHandler的rejectedExecution方法，也就是某种拒绝策略，默认有以下四种。
    • AbortPolicy：当任务添加到线程池中被拒绝时，它将抛出RejectedExecutionException 异常。
    • CallerRunsPolicy：当任务添加到线程池中被拒绝时，会使用提交此任务的线程处理被拒绝的任务。
    • DiscardOldestPolicy：当任务添加到线程池中被拒绝时，线程池会放弃等待队列中最旧的未处理任务，然后将被拒绝的任务添加到等待队列中。
    • DiscardPolicy：当任务添加到线程池中被拒绝时，线程池将丢弃被拒绝的任务。

从线程池为空的状态开始，到来的任务首先与一个新线程一起包装为Worker加入workers集合，如果workers.size达到corePoolSize则加入阻塞队列，如果队长有限，那么队满后加入的任务继续包装为Worker加入workers集合，直到达到maximumPoolSize，此后加入的任务全部采用拒绝策略拒绝

线程状态与线程数量使用一个int存放，高三位存状态，其余位存worker数量

可以看出Worker是内部重要的执行单元，Worker继承了AQS，重写tryAcquire、tryRelease、lock、unlock等方法实现为一个不可重入的排它锁；Worker内部存储一个Thread和一个Task作为任务执行的环境；创建Worker时会利用ThreadFactory创建一个对应此Worker的线程

从构造函数大致就可以看出线程池内部维护的数据结构，下面从submit方法进入看一下内部实现。submit方法调用的是execute方法：

public void execute(Runnable command) {
        if (command == null)
            throw new NullPointerException();

        int c = ctl.get();
        // 小于corePoolSize，为任务创建新线程
        if (workerCountOf(c) < corePoolSize) {
            if (addWorker(command, true)) //以corePoolSize为bound添加worker
                return;
            c = ctl.get();
        }
        // 线程池RUNNING，那么入队
        if (isRunning(c) && workQueue.offer(command)) {
            int recheck = ctl.get();
            // workers 的数量达到了 corePoolSize，任务入队成功，下一瞬间线程池被关闭了，并且没有被阻塞的空闲线程能够立即取出这一任务，那么执行拒绝策略。
            // 是一个非常边界的情况，入队和关闭线程池并发执行
            if (! isRunning(recheck) && remove(command))
                reject(command);
            // 防止线程池中没有线程了
            else if (workerCountOf(recheck) == 0)
                addWorker(null, false);
            // 否则就仅仅只是入队，不为这个task分配新线程
        }
        // offer是非阻塞操作，这里代表队列已满，以maximumPoolSize为bound添加worker，失败则执行拒绝策略
        else if (!addWorker(command, false))
            reject(command);
    }

然后看addWorker中的逻辑

// firstTask：是否为worker分配第一个任务
// core: 是否已corePoolSize作为bound，否则使用maximumPoolSize作为bound
private boolean addWorker(Runnable firstTask, boolean core) {
    retry:
    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c);
        // Check if queue empty only if necessary.
        // 如果线程池已关闭，并满足以下条件之一，那么不创建新的 worker：
        // 1. 线程池状态大于 SHUTDOWN，其实也就是 STOP, TIDYING, 或 TERMINATED
        // 2. firstTask != null
        // 3. workQueue.isEmpty()
        if (rs >= SHUTDOWN &&
            ! (rs == SHUTDOWN &&
               firstTask == null &&
               ! workQueue.isEmpty()))
            return false;
        for (;;) {
            int wc = workerCountOf(c);
            if (wc >= CAPACITY ||
                wc >= (core ? corePoolSize : maximumPoolSize))
                return false;
            if (compareAndIncrementWorkerCount(c)) // 原子修改worker数量
                break retry;
            c = ctl.get();  // CAS失败，检查状态是否被修改，是->跳转到外部循环，否->重复当前循环
            if (runStateOf(c) != rs)
                continue retry;
            // else CAS failed due to workerCount change; retry inner loop
        }
    }
    boolean workerStarted = false;
    boolean workerAdded = false;
    Worker w = null;
    try {
        w = new Worker(firstTask);
        final Thread t = w.thread;
        if (t != null) {
            final ReentrantLock mainLock = this.mainLock;
            // 此类的全局锁，防止与tryTerminate
            mainLock.lock();
            try {
                // Recheck while holding lock.
                // Back out on ThreadFactory failure or if
                // shut down before lock acquired.
                int rs = runStateOf(ctl.get());
                if (rs < SHUTDOWN ||
                    (rs == SHUTDOWN && firstTask == null)) {
                    if (t.isAlive()) // precheck that t is startable
                        throw new IllegalThreadStateException();
                    workers.add(w);
                    int s = workers.size();
                    if (s > largestPoolSize)
                        largestPoolSize = s; // largestPoolSize是线程池维护的一个统计值
                    workerAdded = true;
                }
            } finally {
                mainLock.unlock();
            }
            if (workerAdded) {
                t.start(); // 执行Worker的run方法
                workerStarted = true;
            }
        }
    } finally {
        if (! workerStarted)
            addWorkerFailed(w); // 失败后的清理工作
    }
    return workerStarted;
}

接着走下去看Worker的run方法，调用了外部ThreadPoolExecutor的runWorker方法

final void runWorker(Worker w) {
    Thread wt = Thread.currentThread();
    Runnable task = w.firstTask;
    w.firstTask = null;
    w.unlock(); // allow interrupts
    boolean completedAbruptly = true;
    try {
        // 这里不断地通过getTask从阻塞队列中获得任务执行
        while (task != null || (task = getTask()) != null) {
            w.lock();
            // If pool is stopping, ensure thread is interrupted;
            // if not, ensure thread is not interrupted.  This
            // requires a recheck in second case to deal with
            // shutdownNow race while clearing interrupt
            if ((runStateAtLeast(ctl.get(), STOP) ||
                 (Thread.interrupted() &&
                  runStateAtLeast(ctl.get(), STOP))) &&
                !wt.isInterrupted())
                wt.interrupt();
            try {
                beforeExecute(wt, task);
                Throwable thrown = null;
                // 注意到Runnable.run方法不允许抛出异常，这里接收unchecked异常后重新抛出
                // 异常在Thread没有UncaughtExceptionHandler处理的情况下会导致线程的死亡
                // afterExecute回调可以重新利用抛出的异常
                // Question：不明白什么情况下会接收到Throwable
                // Answer: 我认为有可能子类重写了AbstractExecutorService的newTaskFor方法，
                // 返回了一个自定义的RunnableFuture，其中有可能没有对所有的异常进行收集，
                // 此时这里的catch Throwable就起一个兜底的作用
                try {
                    task.run();
                } catch (RuntimeException x) {
                    thrown = x; throw x;
                } catch (Error x) {
                    thrown = x; throw x;
                } catch (Throwable x) {
                    thrown = x; throw new Error(x);
                } finally {
                    afterExecute(task, thrown);
                }
            } finally {
                // 准备迎接下一个任务
                task = null;
                w.completedTasks++;
                w.unlock();
            }
        }
        completedAbruptly = false;
    } finally {
        // 根据是否abruptly结束（也就是说run方法抛异常导致线程退出还是getTask方法返回null导致退出）
        // 以及当前线程池的状态，completedAbruptly会判断在当前线程死亡后是否还需要重新补充线程
        // 代码就不展开了
        processWorkerExit(w, completedAbruptly);
    }
}

为了解决getTask何时返回null的疑问，看一下getTask()方法的实现

// 注意这里返回null意味着runWorker中线程的退出
private Runnable getTask() {
    boolean timedOut = false; // Did the last poll() time out?

    for (;;) {
        int c = ctl.get();
        int rs = runStateOf(c);

        // 如果等于SHUTDOWN且队列为空 或 大于SHUTDOWN状态，CAS循环减少workerCount，随后线程退出
        if (rs >= SHUTDOWN && (rs >= STOP || workQueue.isEmpty())) {
            decrementWorkerCount();
            return null;
        }

        int wc = workerCountOf(c);

        // Are workers subject to culling?
        boolean timed = allowCoreThreadTimeOut || wc > corePoolSize;

        // 如果线程数量过多（调用setMaximumPoolSize重新设置maximumPoolSize导致）
        // 或者需要计时且超时 && 这不是最后一个线程或队列为空
        // 那么尝试一次CAS，失败则重新刷新状态，成功则线程退出
        // 注意这里可能导致线程池中没有worker且队列空，
        // 如果线程状态合适（不是RUNNING）的话，可能就通过
        // processWorkerExit中的tryTerminate函数终止线程池了
        if ((wc > maximumPoolSize || (timed && timedOut))
            && (wc > 1 || workQueue.isEmpty())) {
            if (compareAndDecrementWorkerCount(c))
                return null;
            continue;
        }

        try {
            // 队列为空时，getTask方法在此阻塞
            Runnable r = timed ?
                workQueue.poll(keepAliveTime, TimeUnit.NANOSECONDS) :
                workQueue.take();
            if (r != null)
                return r;
            timedOut = true;
        } catch (InterruptedException retry) {
            timedOut = false;
        }
    }
}

最后看一下线程池退出的部分，线程池退出一般涉及三个方法

public void shutdown() {
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        checkShutdownAccess();
        advanceRunState(SHUTDOWN); // 设置状态
        interruptIdleWorkers(); // 主要逻辑在这里，就是对每一个空闲线程（通过tryLock判断）调用interrupt方法
        onShutdown(); // hook for ScheduledThreadPoolExecutor
    } finally {
        mainLock.unlock();
    }
    tryTerminate(); // 所有可能导致线程终止的操作都需要执行一次tryTerminate方法
}

public List<Runnable> shutdownNow() {
    List<Runnable> tasks;
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        checkShutdownAccess();
        advanceRunState(STOP); // 设置状态，getTask不再返回任务
        interruptWorkers();  // interrupt所有还在运行的线程
        tasks = drainQueue(); // 导出所有队列中的任务
    } finally {
        mainLock.unlock();
    }
    tryTerminate();
    return tasks;
}

public boolean awaitTermination(long timeout, TimeUnit unit)
    throws InterruptedException {
    long nanos = unit.toNanos(timeout);
    final ReentrantLock mainLock = this.mainLock;
    mainLock.lock();
    try {
        for (;;) {
            if (runStateAtLeast(ctl.get(), TERMINATED))
                return true;
            if (nanos <= 0)
                return false;
            nanos = termination.awaitNanos(nanos); // termination是一个Condition
        }
    } finally {
        mainLock.unlock();
    }
}

// 无界队列
public static ExecutorService newFixedThreadPool(int nThreads) {
    return new ThreadPoolExecutor(nThreads, nThreads,
                                  0L, TimeUnit.MILLISECONDS,

// 单线程+无界队列
public static ExecutorService newSingleThreadExecutor() {
    return new FinalizableDelegatedExecutorService
        (new ThreadPoolExecutor(1, 1,
                                0L, TimeUnit.MILLISECONDS,
                                new LinkedBlockingQueue<Runnable>()));
}

// coreSize = 0，60s空闲线程超时，首次进入的任务会创建Worker，后续任务如果有空闲线程被take阻塞那么正好可以获得，否则创建新线程（建议回头看一下execute函数）
public static ExecutorService newCachedThreadPool() {
    return new ThreadPoolExecutor(0, Integer.MAX_VALUE,
                                  60L, TimeUnit.SECONDS,
                                  new SynchronousQueue<Runnable>());
}

总结

也可以直接参考第一篇参考文章文末的总结和第二篇参考文章，这里借鉴了第二篇参考文章中线程池参数配置的总结，其余部分均为原创

  • 为什么Worker本身也要继承AQS设计为不可重入的排它锁？

    • 一种原因是用于在interruptIdleWorkers函数中判断一个Worker是否空闲，因为在运行时是需要加锁的。
    • 至于为什么不继承ReentrantLock，见StackOverflow上的讨论 （我觉得这个面试肯定不问……）
  • 线程池参数如何设置：简单地讨论需要的线程个数的话，对于CPU密集型任务设为CPU核心数+1，对于IO密集型设为CPU核心数*N，IO耗时越久，这个N可以越大

另外，对于文章开头提出的问题，（终于）我们可以在文末作出解答：

  1. 异步调用是如何实现的？
    首先创建线程池，执行submit(Callable)方法。那么执行submit(Callalbe)操作发生了什么？

    1. 创建一个RunnableFuture类的FutureTask，交由线程池执行
    2. 线程池根据线程状态
      1. 创建新线程执行
      2. 入队响应空闲线程对阻塞队列的take操作或者等待线程空闲
      3. 根据拒绝策略拒绝执行这一任务，默认的拒绝策略有：
        1. 抛异常
        2. 什么也不做，将任务丢弃
        3. 交给调用submit的线程执行
        4. 将队头元素poll抛弃，对此任务执行execute函数

在异步任务计算的同时，调用线程可以做一些别的事情，需要计算结果时调用get方法获得结果。那么调用get获得执行结果时发生了什么？

  • FutureTask.run会调用内部的Callable，将正常调用返回的结果或是异常存储在一个Object对象中，随后通过unpark唤醒所有内部维护的、在等待这个计算结果的线程。
  • FutureTask的run方法会根据结果是否已经产生修改FutureTask的状态，如果状态不合适（比如说结果还没计算出来或者计算还没开始），调用方会让出CPU（yield或者park），同时加入等待队列
  1. 线程池如何管理线程？

  • 线程池中的每个Worker都对应一个线程，当任务到来时

    • if(workers.size < corePoolsize) 创建新的worker
    • else if(worker.size == corePoolsize && queue.notFull) 任务入队
    • else if(worker.size <= maximumPoolSize && queue.full) 创建新的worker
    • else 根据拒绝策略拒绝执行这一任务

  • 对于任务的执行过程无论是否发生异常都不影响方法返回

  • 每个worker都执行一个死循环，不停地调用getTask方法从阻塞队列中获得任务然后执行，worker有两种退出情况
    • 执行过程中抛出异常（在默认情况下FutureTask会接住所有异常从而不会在这一层收到异常）导致线程死亡，另外线程死循环外finally中的processWorkerExit回考虑是否需要在线程死亡后补上一个新线程
    • getTask方法返回null，原因有
      • 线程池状态改变
      • 超过maximumPoolSize
      • 超过corePoolSize但不超过maximumPoolSize的线程空闲太久导致timeout
