Kuberntes client-go informer包源码阅读

vicety · 2021-3-29 · 次阅读

最近项目要写k8s resource watcher，watch指对特定资源的一次订阅请求，订阅后连接保持打开，通过长连接的方式接收来自api-server关于此资源状态变化的推送，相比轮询减轻了api-server的压力。

informer是k8s提供的基于Watch请求包装的一套代码，抛开informer，先看一下如何做一个简单的watch

func main() {
    config, err := clientcmd.BuildConfigFromFlags("", "conf/kube.config")
    if err != nil {
        panic(err.Error())
    }

    clientset, err := kubernetes.NewForConfig(config)
    if err != nil {
        panic(err.Error())
    }

    labelSelector := labels.Set(map[string]string{"k": "v"}).AsSelector().String()

    jobs, err := clientset.BatchV1().Jobs(apiv1.NamespaceAll).List(metav1.ListOptions{LabelSelector: labelSelector})
    if err != nil {
        panic(err.Error())
    }
    resourceVersion := jobs.ListMeta.ResourceVersion

    watcher, err := clientset.BatchV1().Jobs(apiv1.NamespaceAll).Watch(metav1.ListOptions{
        LabelSelector:   labelSelector,
        ResourceVersion: resourceVersion, // 监听指定resourceVersion后发生的变化
    })
    if err != nil {
        panic(err.Error())
    }
    ch := watcher.ResultChan()

    for {
        event := <-ch
        job, _ := event.Object.(*v1.Job)
        fmt.Printf("%v\n", job.ObjectMeta.Name)
    }
}

相当简短，但是有些问题需要考虑

  1. 多线程从resultChan读取，可能出现两个线程处理同一个对象先后不同事件的问题
  2. watcher在api-server默认的超时时间是1800s，需要考虑超时重连
  3. watch返回的resultChan是无缓冲的，如果消费者处理速度小于生产者会阻塞生产者
  4. 在从resultChan读出前，同一对象的先后多个事件应以最后一个为准，因为watcher是level-based的，这样能够提高效率。
    （edge-based相对于level-based更加robust，因为edge有可能丢失）

k8s的informer基于上面这一套简单（但有问题的）代码进行封装，解决了如上问题

看一下如何使用这套框架监听资源变化

// 伪代码，以mpiJob资源举例，资源类型是什么并不重要
func main() {
    informerFactory := externalversions.NewSharedInformerFactory(client.GetK8sResourceClient(), 0)
    mpiJobInformer := informerFactory.Kubeflow().V1().MPIJobs().Informer()
    workQueue := workqueue.NewNamedRateLimitingQueue(workqueue.DefaultControllerRateLimiter(),
                "mpi-job-work-queue"),
    mpiJobInformer.AddEventHandler(cache.ResourceEventHandlerFuncs{
            AddFunc: jw.enqueueEvent,
            UpdateFunc: func(oldObj, newObj interface{}) {
                jw.enqueueEvent(newObj)
            },
            DeleteFunc: func(obj interface{}) {
                var mpiJob *kubeflowV1.MPIJob
                switch obj := obj.(type) {
                case *kubeflowV1.MPIJob:
                    mpiJob = obj
                case cache.DeletedFinalStateUnknown:
                    deletedObj := obj.Obj
                    mpiJob = deletedObj.(*kubeflowV1.MPIJob)
                }
                if mpiJob != nil {
                    jw.enqueueEvent(mpiJob)
                }
            },
        })

    informerFactory.Start(stopCh)
    go wait.Until(runWorker, time.Second, stop)
}

func runWorker() {
    log.Logger.Info("worker start...")
    defer r.HandleCrash()
    for s.processWorkItem() {
    }
}

func (s *JobWatcher) processWorkItem() {
    key, shutdown := s.workQueue.Get()
    defer s.workQueue.Done(key)
    obj, exists, err := mpiJobInformer.GetIndexer().GetByKey(key.(string))
    mpiJob := obj.(*v1.MpiJob)
    // do things based on mpiJob status
}

与第一版相比，省略了client初始化，加上了一些通用的对象处理逻辑，总体上用户需要关心的代码量减少了

下面介绍informer包中几个重要的数据结构，可以配合下方的类关系图理解

  • informerFactory: 可以通过builder模式，以group version kind的形式创建资源类型的单例informer，相对于手动创建informer减免了listFunc和watchFunc的编写（但是也不支持对list/watch请求进行配置，如指定label selector），在有多个watcher需要创建informer时能够利用单例。

  • sharedIndexInformer: 用于对特定某种资源类型进行list&watch，并根据watch到的事件调用用户注册的对应处理函数。informer的主要组件包括controller、indexer、processor

    • controller: 用于对特定资源类型进行list&watch，并将watch到的结果发往一个队列（DeltaFIFO），同时更新本地缓存（indexer）中此对象的最新状态，此外，另一个线程不断消费队列，发往processor组件的addCh。controller的主要组件是reflector

      • reflector: 负责进行list&watch，并将结果写入队列（DeltaFIFO），主要成员包括listWatcher（list和watch的函数指针，不走informerFactory创建时可配）和store（也就是DeltaFIFO）

        • store: DeltaFIFO对象，一个特殊的队列，下文介绍，同时持有indexer的引用

    • processor: 包括addCh和nextCh两个无缓冲Channel，addCh接收来自队列DeltaFIFO的输出，存入一个可扩容的ringBuffer，并不断从ringBuffer取出放入nextCh，另一个线程从nextCh消费，根据资源事件类型（创建/更新/删除）调用用户注册的对应handler

    • indexer：watcher到的资源会更新到indexer中，下文解释作用

设计这些组件的目的是为了解决先前直接watch可能出现的问题：

  1. 通过DeltaFIFO实现
  2. informer包的list&watch放在一个死循环中实现重试
  3. workQueue和ringBuffer都可以起到缓冲作用，这里还不太清楚使用ringBuffer的优势在哪里
