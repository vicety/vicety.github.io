The Dataflow Model: A Practical Approach to Balancing Correctness, Latency, and Cost in Massive-Scale, Unbounded, Out-of-Order Data Processing

wallhaven o59gvl

对大数据了解不多，文章的理解可能不全正确

贡献：

  • windowing model，支持 unaligned event-time windows
  • triggering model
  • 增量的处理模型
  • 认为 unbounded datasets 的处理可以看做通过多次 batch 操作实现
  • 提供与下层执行引擎（batch or stream）无关的编程模型
  • 定时 punctuation 等价于 micro-batch，仍然有时延问题，引入 trigger 做数据驱动的 window emit
  • 整合仅支持 streaming 的 MillWheel 和仅支持 batching 的 FlumeJava 暴露一套流批一体的 API

常见的几种窗口：

  • aligned window：应用于窗口时间内的所有数据
  • unaligned window: 应用窗口时间内的部分数据（如 per-key 的）
  • fixed windows / tumbling windows: 定长、通常 aligned、不重合的窗口
  • sliding windows: 定长、定滑动距离、通常 aligned、存在重合的窗口
  • session windows: 通常在一个 timeout 内的、相同 key 的事件 group 到同一个窗口，通常是 unaligned 的

两个时间概念：

  • event time: 事件在事件源的发生时间
  • processing time: 从我们的系统中观察到此事件的时间

processing time 总是大于等于 event time

经典 batch 模型对于 (k, v) 数据的转换提供两个核心 API：

  • ParDo (ParallelDo)：对应 map 或 flatMap
  • GroupByKey：顾名思义

对于 unbounded data，ParDo 照样处理，但对于 GroupByKey，我们不知道何时数据结束（可以开始 group），常见的办法是引入 windowing
支持 group、window 的系统通常都将 GroupByKey 实现为 GroupByKeyAndWindow
对于任何 windowing 策略，通常都涉及两个 API：

  • AssignWindows：将数据加入某些 windows，需要复制一份数据
  • MergeWindows：合并 windows

为了实现 event-time windowing 数据使用 (k, v, event_time, window) 四元组表示，新来的数据默认被分给 global window
文章之后介绍了 GroupByKeyAndWindow 的实现：

  • 默认已经被 assign 给某个 window
  • 丢弃时间戳（其作用是 assign window，在前一步已经做了）
  • 合并相同 key 的 event（k1 v1 w1, k1 v2 w2 -> k1[v1w1, v2w2]）
  • 按 key 合并 window（k1[v1w3, v2w3]）
  • 合并相同 window 的 value（k1w3[v1, v2]）
  • 恢复到四元组格式 (k1, [v1,v2], w3.end, w3)

Question: 原文保证 wm 的正确性的这句话是为什么？

we set the timestamp to the end of the window, but any timestamp greater than or equal to the timestamp of the earliest event in the window is valid with
respect to watermark correctness.

创建窗口后还需要 emit 窗口，通常使用 watermark 来推理 window 是否 complete，但有如下问题：

  • wm 太快：可能有迟到的数据
  • wm 太慢：如果我们等待迟到数据太久，那么会伤害实时性

Lambda Architecture 在 streaming 中采用引入最低时延的 watermark 策略，通过稍后执行的 batch 处理实现最终的正确性，但这需要维护两套（stream 和 batch）pipeline。论文引入 trigger 的概念，尝试使用一套 pipeline 解决上面提到的 window 过快/过慢 的问题

  • windowing 决定 event time 维度上 数据会被如何分组
  • triggering 决定 process time 维度上 group 后的数据何时被 emit

相同 window 可能发生多次 emit，Dataflow 提供如下几种 emit 数据间的关系：

  • discarding：window emit 后清空 buffer，适用于下游希望多次 emit 的数据是独立的情况
  • accumulating：emit 后不清空 buffer，下次 emit 视为覆盖上次 emit
  • accumulating & retracting: 当多个 groupByKeyAndWindow 算子相连接时，上游多次 emit 的 window 可能在下游被 window merge 到不同的 window（原文是说在下游可能会被 group 到不同的 key，不太理解，也许是下游采用了不同的 key 视角？），因此需要先回退之前的 emit，再 apply 新的 emit，实现上维护上次 emit 的值即可

文章给出了一系列 Dataflow Model 的 output pattern 作为 example（疑惑 example 中给的 watermark 是按什么规则画的，不过不影响理解其他部分）

  1. 开始以经典 batch 系统作为后续 examples baseline，在 streaming 系统中相当于 global window，在收到所有数据后 emit 一次，wm 可以看做从负无穷跳变到正无穷
  2. 例1 要求数据是 bounded 的，对于 unbounded 的情况，Dataflow 的默认 triggering 是当 wm 超过 window 结束边界时 emit window，但这一规则对于无界的 global window 是无效的，Dataflow 提供了 processing time 维度的定时 trigger API
  1. 对于不使用 global window 的情况（其实就是 streaming 的场景），可以配置 event time 维度的 window 划分，上面提到过，默认的 trigger 策略是当 wm 越过 window 结束边界时 emit。文章给出了 window + batch / micro-batch / streaming 的 API
  2. 对于 window + streaming、对于给定的 wm 曲线，发现 streaming 的实时性相比于 micro-batch 下降了（矩形高度相对于 micro-batch 的变高了），Dataflow 也支持在 streaming 模式下添加基于 processing time 的定时 trigger 来提高实时性

  1. 最后给了一个复杂的例子，采用 session window，window 长度一分钟，如果与其他 window 重合，那么会合并，同时这个例子采用 accumulate & retract 的策略
