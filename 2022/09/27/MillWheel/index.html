MillWheel: Fault-Tolerant Stream Processing at Internet Scale

看完文章后最大的问题其实是 “水位线是如何生成的”，参考[1]大致得到了解答

概述

  • 文章贡献：提出了一个流处理的编程模型，应用到生产环境并进行分析
  • 用例：Google Zeitgeist，用于获得实时的搜索趋势（也包含某关键词热度的异常下降，文章中叫 dipping）。对每分钟的搜索数据进行聚合，与模型预测进行对比，同时将聚合数据喂给模型使其更新。
  • 使用低水位（low watermark）这一概念表示系统可以接受 “先于 lwm 的任何消息都不会再到达” 这一结果
    • 或者可以理解为，允许乱序 ingest，使得系统实现简单，但需要使用 lwm 尽量减少乱序带来的问题（尽量确保 lwm 以前的数据到齐）
  • 提供 exactly-once
  • 保证不丢数据
  • 提供良好的扩展性

系统设计

  • 文章中叫 computation，实际上就是 DAG 中的节点，节点的输入输出格式为 (key, value, timestamp)

    • 对每个 input stream 可以配置 Key Extractor 从输入中提取 key，根据业务需要选择这个 key，如 Query。也可以对同一个 stream 使用不同的 key extractor，当成多个 stream 来用
    • timestamp 字段也是由用户指定的（除 source，我的理解）

  • 每个 computation 中的用户代码能够访问到自己的 per-key state，这对实现按 key 聚合的操作是很有用的。单 key 数据的处理是串行的，但不同 key 的处理可以并行

  • 状态的存储也是 per-key 的，后端是 Bigtable 或 Spanner

  • 水位线

    • （非 source node 的）水位线等于 min(cur_node.lwm, upstream_node))，而当前节点的 lwm 是根据当前节点尚未处理完成（未收到下游 ACK）的最早 event 的时间戳决定的。从表达式也可以看出下游需要订阅上游节点的 lwm 变化事件
    • source node 的水位线是通过 injector 实现的，论文给出了一个例子：某个 injector 正在读一些文件，那么可以将 lwm 设置为这些文件的创建时间中最早的那个（假设使用文件的创建爱你时间作为 event timestamp，这使得下游基于 lwm 的 timer 能够等到所有文件全部处理完后才触发 handler（开始聚合））
    • node 由于事件到来而更新水位线后，会将这一更新通知中心化的 central authority 做持久化，（我的理解）node 的 lwm 更新后会将更新后的值同步到订阅的下游节点
    • 对于迟到的数据根据配置可以丢弃或单独处理，但无论如何 lwm 总是单调递增的
    • 用户在对产生的 event 赋的 timestamp 应当不早于这个 event 基于的 source event 的最大 timestamp
    • 关于分片服务上报的水位线如何影响整个 DAG node 的水位线见下图（不是论文的内容，来自 Flink）

  • Timer：根据 wall time 或 lwm 触发的一个 handler，比如说检测 spike 不用等完一个窗口，但检测 dip 需要等完一个窗口，再根据窗口内的数据数量与模型预测比较判断是否是一个 dip，这就用到了 Timer

  • 算子间通过 RPC 通信

Fault Tolerance

  • exactly-once
    • 产生 event 时会对其赋 UID 用于判重
    • node 收到的事件会先根据内存中维护的布隆过滤器 fast path 判断是否是新事件（filter miss），对于 hit 的情况则需要查存储判断是新数据还是旧数据。当 node 可以确定所有 sender 都不再 retry 时（猜测是 retry 有最大时间），可以在 Bloom Filter 中 GC 这个数据（Bloom Filter 有版本是支持删除操作的）
    • 如果是新数据，处理后，可能发生变化的 timer、state、production（即输出的 event）需要合并做一次原子持久化
    • 随后向上游 ACK（上游未收到会永久重试）
    • 随后向下游发送
  • strong-production：在向下游发送前，对状态（timer、state、production）变化做原子的持久化。考虑如下场景：向下游发送后没有持久化，尚未收到下游 ACK，挂掉，重启后向下游重新发送，但发送前到来了这个窗口内的新数据，导致向下游发送的数据有变，下游无法简单地判断它是否是新数据。这就是为什么向下游发送前需要持久化
  • weak-production and idempotency：如果业务层不需要exactly-once、strong-prediction（业务层有相关机制，或业务层本身没有 strong-production中说的问题，并且是幂等的），也可以关掉这些特性。前者在 ingest event 时不过滤即可，后者在向下游发送 event 前不做持久化即可。但为了保证数据不丢，仍然需要收到来自下游的 ACK 后，才能回复给上游一个 ACK，这相当于把整个 DAG 偶合起来了，端到端的 event 传递会放大错误概率与时延。虽然减少了同步落盘，反而使得性能变差。解决方案是，如果向下游的请求 pending 太久，那么还是会做一次持久化，持久化后就可以不管下游是否 ACK，而直接向上游回复 ACK，从而解除上游阻塞，释放资源。
  • 状态管理：
    • user state、timers、production checkpoints（就是上面提到的 outbound event 持久化）这些单个 event 导致的状态变化的持久化需要被包装为一个原子操作
    • 由于 load balance、failure，算子以及状态可能被迁移到其他机器上去运行，带来的一个问题是，旧机器（比如假设它与存储以外的网络断开）可能仍然在写存储，导致脏写问题，或者新机器的内存 cache 与存储不一致的问题。MillWheel 采用 lease 机制确保任何时间段只有一台机器允许写入。上文提到的 lwm 更新也需要类似机制确保 single writer[2]

flink 实现的各种 watermark
https://github.com/apache/flink/tree/master/flink-core/src/main/java/org/apache/flink/api/common/eventtime

  • BoundedOutOfOrdernessWatermarks
    维护见过的事件的最大时间戳，emit 的时间戳等于 maxTimestamp - outOfOrdernessMillis - 1
  • AscendingTimestampsWatermarks
    等价于 BoundedOutOfOrdernessWatermarks 的 outOfOrdernessMillis 取 0
  • CombinedWatermarkStatus
    对分片的 watermark 做合并
